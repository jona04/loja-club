# Loja Club — Backend

FastAPI **modular monolith**: SQLModel + Alembic over PostgreSQL, Redis for
cache/locks/queue, an **arq** worker for background jobs, and AWS S3/CloudFront
for object storage. Multi-tenant by `store_id` on every commercial table.

## Layout

```text
app/
  core/            config, db, security, api helpers, money, storage, queue
  db/base.py       shared SQLModel mixins (UUID, timestamps, soft delete, store_id FK)
  modules/<name>/  one module per domain:
                     models.py · enums.py · schemas.py · services.py
                     repositories.py · routes.py · permissions.py · tasks.py
  api/             router aggregation + shared deps
  alembic/         migrations
  models_registry.py   imports every module's models (metadata for Alembic/tests)
tests/             unit + integration (pytest)
scripts/           lint.sh · test.sh · tests-start.sh · prestart.sh · format.sh
```

Convention: `models.py` holds only `table=True` tables + their `*Base`;
`schemas.py` holds API DTOs; `enums.py` holds enums. See [`CLAUDE.md`](../CLAUDE.md)
and [`docs/04`](../docs/04_fastapi_template_adaptation.md).

## Prerequisites

- [Docker](https://www.docker.com/) (for Postgres, Redis, the worker, …)
- [uv](https://docs.astral.sh/uv/) for Python env/dependencies

## Setup

```bash
uv sync                                 # create .venv and install deps
docker compose up -d --wait db redis    # infra for host-run code/tests
```

The host ports are non-standard (Postgres `5442`, Redis `6399`) so the stack
won't clash with other local services. Anything you run on the host against the
stack must point at those ports — hence the `POSTGRES_PORT=5442 REDIS_PORT=6399`
prefix below. `uv run <cmd>` runs inside the project venv without activating it.

## Running

- **Full stack (recommended):** `docker compose up -d` from the repo root (API at
  http://localhost:8800, docs at http://localhost:8800/docs).
- **Hot reload in a container:** `docker compose watch`.

## Tests & coverage

Integration tests run against the real dev Postgres; each test runs in a
transaction that is rolled back (isolated, order-independent).

```bash
POSTGRES_PORT=5442 REDIS_PORT=6399 uv run coverage run -m pytest tests/
POSTGRES_PORT=5442 REDIS_PORT=6399 uv run coverage report --fail-under=90
# single file / -x to stop on first failure:
POSTGRES_PORT=5442 REDIS_PORT=6399 uv run pytest tests/integration/test_catalog_models.py -x
```

## Lint gate

```bash
uv run bash scripts/lint.sh     # mypy app tests · ty check app · ruff check · ruff format --check
```

Run it through `uv run` so the venv is active (the script calls bare
`mypy`/`ruff`). CI runs the same gate. Auto-fix formatting with
`uv run bash scripts/format.sh`.

## Migrations (Alembic)

Models are registered in `app/models_registry.py`, so autogenerate sees them.

```bash
POSTGRES_PORT=5442 REDIS_PORT=6399 uv run alembic upgrade head
POSTGRES_PORT=5442 REDIS_PORT=6399 uv run alembic revision --autogenerate -m "create X"
POSTGRES_PORT=5442 REDIS_PORT=6399 uv run alembic check     # must report "No new upgrade operations"
```

Review autogenerated files: order `create_table` by FK dependency, keep partial
unique indexes (`postgresql_where`), and **name foreign keys explicitly** (a
`None` name breaks `downgrade`). Commit the migration with the model change.

## Worker (arq)

Background jobs run **off the request**. Code enqueues only through
`app.core.queue.enqueue(...)`; tasks are registered in
`app.core.queue.WorkerSettings.functions`. The `worker` service runs them:

```yaml
# compose.yml
worker:
  command: arq app.core.queue.WorkerSettings
```

It uses the **same image as the backend**, so after adding/changing a task,
rebuild it:

```bash
docker compose up -d --build worker
```

Example flow (image upload): the request stores the original on S3, creates the
`media_files` row (`status=processing`) and calls
`enqueue("generate_image_variants", media_id)`; the worker later downloads the
original, builds the thumbnail/card/product/zoom variants and flips the row to
`ready`.

### Inspecting the queue

arq stores everything in Redis under the `arq:` prefix. Connect **Redis Insight**
(or `redis-cli`) to `localhost:6399` (no password in dev) and look for:

| Key | What it is |
|---|---|
| `arq:queue` | the pending queue (sorted set); member count = jobs waiting |
| `arq:queue:health-check` | **human-readable** worker status (the one to watch) |
| `arq:job:<id>` | a queued job's payload (serialized — shows as binary) |
| `arq:result:<id>` | a finished job's result (expires via TTL) |
| `arq:in-progress:<id>` | a job currently being processed |

```bash
# queue length and live worker status
docker compose exec -T redis redis-cli ZCARD arq:queue
docker compose exec -T redis redis-cli GET arq:queue:health-check
# → "Jun-08 14:07:07 j_complete=0 j_failed=0 j_retried=0 j_ongoing=0 queued=0"

# follow it live
watch -n1 'docker compose exec -T redis redis-cli get arq:queue:health-check'
```

arq has **no official web dashboard** (unlike Celery's Flower). The
`arq:queue:health-check` key and `arq --check app.core.queue.WorkerSettings` are
the built-in options; job/result payloads are pickled, so only the health-check
key is plain text.

## Shared external clients (INV-F6)

External clients are created **once and reused** — never per call. Each lives in
its `app/core/*` module (so the `worker`, which has no FastAPI app, can use them
too). The **app lifespan** ([`app/main.py`](app/main.py)) is the **single
shutdown point** that releases them all (the worker releases its own on its
shutdown).

| Client | Module | Accessor | Lifecycle |
|---|---|---|---|
| DB engine | `core/db.py` | `engine` → a `Session` per request (`get_db`) | created at import · `dispose_engine()` on shutdown |
| Redis | `core/cache.py` | `get_redis()` | created at import · `close()` on shutdown |
| S3 (boto3) | `core/storage.py` | the `storage` functions | lazily cached · `close_client()` on shutdown · `reset_client()` in tests |
| arq pool | `core/queue.py` | `enqueue()` | lazily created · `close_pool()` on shutdown |
| HTTP (httpx) | `core/` (added when first needed) | a shared `httpx.AsyncClient` | lazily created · closed on shutdown |

Per request you take **one unit of work** (a `Session`, a Redis command, an HTTP
request) — never a new client. Domain code never instantiates its own client.

## Email

All email is **enqueued to the worker** (task `send_email`), never sent inline
(INV-F5) — render in the worker, retry there. Local emails land in **Mailcatcher**
(http://localhost:1090); SES/SMTP is wired in the deploy phase.

## Object storage (AWS)

`app/core/storage.py` wraps S3 (boto3); the bucket is private and public objects
are served via CloudFront. Config in `.env` (region default `us-east-2`):
`S3_REGION`, `S3_BUCKET`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`,
`CDN_BASE_URL`.

- With the AWS vars **empty**, storage tests run against **`moto`** (mocked S3).
- With **real dev credentials**, the env-gated **smoke tests** also run (real S3
  upload + a CloudFront GET). Never commit those credentials.
